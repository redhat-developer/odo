## So what's new?

odo v3 comes with a new set of commands that replaces the old ones but essentially do the same thing, in a faster and efficient way.

#### Personalize Devfile configuration
`odo init` also provides an interactive way to configure your Devfile according to your project requirement. With this feature, you can modify the environment variables exported to and ports exposed by your application on the cluster.

What sets this apart from `odo push` of odo v2?
odo v3 combines the `odo push` and `odo watch` commands from odo v2 into a single `odo dev` command.

Additionally it also provides,
* Continuous workflow
* Manual trigger to apply local changes on the cluster
* Resource cleanup
* Better watch system


#### Continuous workflow
`odo dev` is a non-terminating command that runs until user hits Ctrl+c. It continuously watches the directory for any new changes, and automatically syncs them with the application running on the cluster. 

#### Manual trigger to apply local changes on the cluster
By default, `odo dev` automatically detects any new change in the directory (this includes any changes that occur to the Devfile), and syncs it with the application running on the cluster, but it is possible to do this trigger manually by running the command with a `--no-watch` flag and pressing `p` to trigger the sync.

#### Better watch system
`odo dev` continuously monitors the resources created to run the application on the cluster, and ensures that the application always stays running.

#### Resource cleanup
Before exiting, `odo dev` will cleanup any resources that were created for running the application on the cluster.


### odo add binding

`odo add binding` allows the application to connect to an Operator Backed service.

```shell
$ odo add binding
? Do you want to list services from: current namespace
? Select service instance you want to bind to: cluster-sample (Cluster.postgresql.k8s.enterprisedb.io)
? Enter the Binding's name: my-node-app-cluster-sample
? How do you want to bind the service? Bind As Files
? Select naming strategy for binding names: DEFAULT
 âœ“  Successfully added the binding to the devfile.
Run `odo dev` to create it on the cluster.
You can automate this command by executing:
  odo add binding --service cluster-sample.Cluster.postgresql.k8s.enterprisedb.io --name my-node-app-cluster-sample
```

What sets it apart from `odo link` command of odo v2?
* Interactive mode
* Ability to connect to a service in any namespace
* Run from anywhere
* Does not require a running application
* Only supports Bindable Operators
* Requires Service Binding Operator

#### Interactive mode
`odo add binding` provides a very nice interactive mode that lists bindable service across all the namespaces, and provides a way to personalize related configuration, making the experience nicer.

#### Ability to connect to a service in any namespace
`odo add binding` now allows the application to connect to any bindable service in any namespace, it doesn't necessarily have to be in the same namespace as the application.

#### Run from anywhere
`odo add binding` can be run from anywhere and does not require access to a Devfile in order to work. With this method, you can either directly create the Service Binding on the cluster, write the YAML to a file, or simply display it.

#### Does not require a running application
As opposed to `odo link` of v2, `odo add binding` does not require for the application be running on the cluster, making it faster.

#### Only supports Bindable Operators

As opposed to v2, where `odo link` could link an application to another application and any Operator Backed service, as of now, v3 only supports connecting to services that are backed by Operators that are considered [bindable](https://github.com/redhat-developer/service-binding-operator#known-bindable-operators) by the Service Binding Operator.

Based on the user feedback, we might work on adding these features back, but as of now they are not implemented.

Regardless, there are workarounds to connect to another application
or non-bindable service, read [this blog](https://odo.dev/blog/binding-external-service-with-odo-v3) to know how.

#### Requires Service Binding Operator

`odo add binding` requires that the Service Binding Operator is installed before it is run. The command relies on the operator to provide it with a list of Bindable services.

This will be fixed in the future releases.

You might find yourself wondering how `odo dev` is different from `odo deploy` and when should you use the either one, you can read [this article](https://odo.dev/docs/overview/dev_and_deploy) to get an answer that question.


### odo registry

`odo registry` now lists all the devfiles available by the devfile registries that are added to your system.

How is it different from `odo registry` command of odo v2?
`odo registry` of v2 was only useful in creating/deleting/updating/listing Devfile registries, however in v3, this command can be used to list the devfiles.

Additionally, it also provides:
* Filtering the devfiles
* Detailed information about a devfile

#### Filtering the devfiles
It is possible to filter the list of devfiles based on a keyword(`--filter`), a devfile registry(`--devfile-registry`), or the devfile name(`--devfile`).

#### Detailed information about a devfile
You can get detailed information about a devfile by running the command with `--details` flag.
